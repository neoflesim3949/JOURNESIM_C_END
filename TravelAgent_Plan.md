# 旅行社服務專區（Travel Agency Service Zone）計劃

> 狀態：討論定案，採「**UI 先行**」——先用假資料把畫面/流程做出來，確認後再定後端 schema 與 API。

## 一、功能概述

後台新增一個大模組「**旅行業專區**」，含兩個子模組：

- **人員管理**：建立旅行社的「業務/現場人員」帳號，讓他們登入查看；之後接一個前端讓現場人員發卡給旅客。
- **出團管理**：旅行社建團 → 設團員名單 → 設該團基礎方案 → 每位團員拿到專屬連結 → 團員自行決定改 eSIM / 升級方案。

## 二、已定案的決策

| 項目 | 決定 |
|---|---|
| 多租戶 | **要**。有「旅行社」這層，各社有自己的業務帳號、只看自己的團。 |
| 團員動作 | 選完 + **線上付款**；旅行社出團前統一檢查、**按鍵發放**（付款在選擇時、開卡延到發放）。 |
| 金流 | 團員自己**線上刷卡**；系統計算**差額利潤**結算給旅行社。 |
| 利潤算法 | **只算線上實際收款（升級差額）的利潤**。基礎方案成本結算走線下，本系統不碰。 |
| 發放 = 開卡 | **即時開卡**：eSIM 打 BC `F040` 產 QR；SIM 從旅行社庫存指派 ICCID。 |
| 人員角色 | **兩層**：管理者（建團/設價/管人員）＋ 業務（查看/發放）。 |
| 基礎方案是否免費 | 旅行社**自行設定**，**每團一個開關**（`base_is_free`）。 |
| 建議售價來源 | ＝該套餐在 `/admin/packages` 的售價。旅行社售價可往下調、**不得高於**建議售價（系統擋上限）。 |
| 基礎方案粒度 | 通常是 `/admin/packages` 的一個方案；可各選一個 **SIM** + 一個 **eSIM**。SIM/eSIM 互換免費（都算基礎），超過基礎才收差額。 |
| 卡片庫存 | 發卡的 ICCID **必須掛在該旅行社的庫存**。需要「FLESIM 把卡片分配給旅行社」的機制。 |
| 發卡方式 | 兩種：①**統一發**（出團前先輸入卡號）②**現場發卡**（現場人員點名單、輸入或掃描卡號、儲存＝發卡）。 |

### 金流公式（每位團員）
```
線上實收 = base_is_free ? max(旅行社售價_所選 − 旅行社售價_基礎, 0)   // 減少不退 → 最低 0
                        : 旅行社售價_所選(全額)
旅行社利潤 = 線上實收 − 我方成本對應部分
我方成本 = BC 結算價換算 TWD（沿用現有 shopee-pricing 算法）
```
- 錢由團員線上刷給 FLESIM，FLESIM 之後把「利潤」結算給旅行社。

### 待定的實作細節
- **實體 SIM 的「發卡」**：先假設＝從旅行社庫存指派 ICCID ＋ 記錄（掃碼/輸入），不另打 BC；eSIM 才走 F040。若實體卡需在發放時由 BC 載入方案內容，留待 Phase 3 定。

## 三、三個介面

1. **FLESIM 後台** `/admin/travel`：建立/管理旅行社、設我方成本、分配卡片庫存給旅行社、看結算報表。
2. **旅行社後台** `/travel/*`（業務登入，新的一套帳號）：建團、加團員、拉方案設售價、設基礎方案、出團前檢查＋發放、人員管理。
3. **團員專屬連結** `/tour/[token]`：免登入，看基礎方案、選 eSIM/升級、刷卡付差額。

- `manual_iccids` 加 `agency_id`（卡片分配給哪個旅行社；發卡的 ICCID 必須屬於該社）
- 結算：★ 可用檢視表或 `travel_settlements`（agency_id、月份、線上實收、我方成本、應結算利潤、settled）— 由訂單自動彙總
- 套餐涵蓋國家：★ 沿用 /admin/packages 既有的多國資料；**精準匹配＝套餐須涵蓋團選的所有國家**（superset）

## 七、技術備註
- 旅行社/業務登入是**全新帳號層**（現況只有單一 admin 密碼 + Supabase 顧客會員）。密碼用 Node `crypto` scrypt 雜湊，不加新套件。
- 團員連結是**免登入、憑 token**的新模式（現況無此 pattern）。
- 可沿用：`shopee-pricing.ts`（成本/售價算法）、TapPay checkout、manual_iccids 卡庫、BC F040/F010。
